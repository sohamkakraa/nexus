# Nexus zero-cost launch campaign

## Positioning

**Category:** local-first AI Council for macOS  
**Promise:** Two AI models. One considered answer.  
**Audience:** researchers, developers, founders, analysts, writers, and privacy-conscious AI power users  
**Proof:** open source, Keychain-backed BYOK, visible Council protocol, local history, explicit tool approvals  
**Avoid:** “unlimited,” “fully private,” “zero data leaves your Mac,” “autonomous employee,” or unsupported model names

## Public identities

Preferred handle: `@NexusCouncil`  
Fallbacks: `@GetNexusAI`, `@NexusForMac`

Use the same display name, icon, bio, link, and banner on GitHub, X, Bluesky, LinkedIn, YouTube, Mastodon, and Product Hunt. Account creation and posting must use platform-native authentication; no credentials belong in this repository.

**Bio:** `Two AI models. One considered answer. Free, open-source, local-first AI Council for macOS.`

**Link:** `https://nexus.sohamkakra.com`

## Launch assets

- Hero line: `Don't ask one AI. Convene a council.`
- Short demo: 30–45 seconds, no private content or real API keys
- Screens: Council answer, file context, research sources, permission prompt, voice session
- Proof links: source, security architecture, privacy notice, release checksums
- CTA: `Download for macOS` only after a downloadable, tested release exists; before that use `Build the source beta`

## Four-week campaign

### Week 1 — Build trust

1. Publish the repository, security model, privacy notice, roadmap, and reproducible release workflow.
2. Share an architecture post: “Why two models are better only when they can disagree.”
3. Post a 20-second Council interaction with fictional data.
4. Invite ten macOS AI power users to a public beta issue—not a private mailing list.

### Week 2 — Show work

1. Ship daily use-case clips: research, code review, strategy, document critique, meeting transcript.
2. Publish a benchmark methodology comparing Solo and Council answers without cherry-picking.
3. Submit to Hacker News Show HN, Reddit communities where self-promotion is permitted, and relevant open-source directories.
4. Ask contributors for accessibility, security, and packaging reviews.

### Week 3 — Earn distribution

1. Launch on Product Hunt only after notarized downloads are ready.
2. Publish a guest post or dev.to article explaining the capability broker.
3. Create a “Council prompt” public collection via GitHub Discussions.
4. Invite MCP server maintainers to document safe Nexus compatibility.

### Week 4 — Compound

1. Publish release notes and transparent metrics: stars, downloads, issues resolved, crash-free beta sessions if users explicitly report them.
2. Turn repeated public questions into docs and FAQ schema.
3. Highlight contributor work.
4. Select one evidence-backed feature from public issues for the next milestone.

## Success metrics

All metrics are aggregate and public-platform-native; the app remains telemetry-free by default.

- GitHub stars, forks, contributors, issue response time
- Release downloads and checksum fetches
- Vercel aggregate page views and Core Web Vitals
- Documentation search impressions and rich-result validity
- Product Hunt/Hacker News/Reddit engagement
- Opt-in, reviewed feedback packages

## Guardrails

- Never upload user prompts, chats, files, recordings, model responses, API keys, or local diagnostics.
- Never fabricate testimonials, users, benchmark wins, or security claims.
- Respect community self-promotion rules and rate limits.
- No paid ads, bought followers, cold-DM automation, scraping, or engagement bots.
- Social posts may be scheduled after account owners approve platform access and content; credentials remain in each platform’s secret store.
